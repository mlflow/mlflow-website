The recent LiteLLM supply chain compromise demonstrated why pinned, locked dependencies matter for ML models. In March 2026, attackers published backdoored versions of `litellm` to PyPI after stealing a publish token through a compromised GitHub Action. The malicious packages harvested SSH keys, cloud credentials, and API tokens from every machine that installed them. Projects using `pip install litellm` without version pinning pulled the compromised release automatically. Projects with a lockfile that pinned the previous version were not affected.

MLflow, with over 33 million monthly PyPI downloads, now integrates with [uv](https://docs.astral.sh/uv/), the fast Python package manager from [Astral](https://astral.sh/). When you work in a uv-managed project, MLflow automatically uses your lockfile to capture the complete, hash-verified dependency graph. No extra parameters, no manual `uv export`, no glue code.

## What uv lockfiles give you

Before this integration, MLflow inferred model dependencies by inspecting the packages imported during `log_model()`. This approach has known gaps:

- **Missing transitive dependencies.** If your model depends on `pandas`, which depends on `pytz`, but `pytz` was never imported directly, it might not appear in `requirements.txt`.
- **Environment bleed.** Dev tools, test frameworks, and unrelated packages from your virtualenv leak into the captured requirements.
- **No platform markers.** `pip freeze` strips PEP 508 environment markers, so platform-specific dependencies break on different operating systems.
- **No integrity verification.** Version pinning alone does not prevent tampered packages. An attacker who overwrites a pinned version on PyPI bypasses version checks entirely.

uv lockfiles solve all four. A `uv.lock` file captures the complete transitive dependency graph with exact versions, platform markers, and SHA-256 hashes for every package wheel. When `uv sync` installs dependencies, it verifies each downloaded package against its recorded hash. If an attacker publishes a modified package under an existing version number, the hash check fails and installation is blocked.

```
# Inside uv.lock, every package has a pinned hash:
[[package]]
name = "litellm"
version = "1.82.6"
source = { registry = "https://pypi.org/simple" }
wheels = [
    { url = "...", hash = "sha256:abc123..." },
]
```

When MLflow captures your `uv.lock` as a model artifact, it preserves these hashes. Any model restored with `uv sync --frozen` will verify package integrity before installation.

## Zero-config auto-detection

If your project already uses uv, there is nothing to change. MLflow detects the presence of `uv.lock` and `pyproject.toml` in your working directory and handles everything automatically:

```python
import mlflow

# Inside a directory with uv.lock + pyproject.toml
mlflow.pyfunc.log_model(
    name="my-model",
    python_model=my_python_model,
    input_example=sample_input,
)
```

Behind the scenes, MLflow:

1. Detects `uv.lock` and `pyproject.toml` in the current directory
2. Runs `uv export` to produce a pinned requirements list from the lockfile
3. Writes the result to `requirements.txt` in the model artifact
4. Copies `uv.lock`, `pyproject.toml`, and `.python-version` (if present) as model artifacts

The saved `uv.lock` enables exact environment restoration via `uv sync --frozen` when the model is loaded later. If uv is not available or `uv sync` fails at restore time, MLflow raises an error rather than silently falling back to pip. This ensures the hash-verified lockfile is always used and prevents unpinned dependencies from reaching production.

## Monorepos and explicit paths

In a monorepo where your training script lives in a different directory from your `uv.lock`, use `uv_project_path` to point MLflow to the right location:

```python
mlflow.pyfunc.log_model(
    name="my-model",
    python_model=my_python_model,
    uv_project_path="/repo/services/ml-training",  # Where uv.lock lives
)
```

This is useful when your repository has multiple services with separate lockfiles, or when your training job runs from a subdirectory that does not contain the project root.

## Dependency groups and extras

Production systems typically need a much tighter set of dependencies than local development environments. Your training environment needs `torch` and `xgboost`, your serving environment needs `flask` and
`uvicorn`, and your dev environment needs `pytest` and `ruff`. MLflow lets you selectively include specific [dependency groups](https://docs.astral.sh/uv/concepts/projects/dependencies/#dependency-groups) and [optional extras](https://docs.astral.sh/uv/concepts/projects/dependencies/#optional-dependencies) when exporting from the lockfile:

```python
mlflow.pyfunc.log_model(
    name="my-model",
    python_model=my_python_model,
    uv_project_path="./",
    uv_groups=["ml"],           # Include the "ml" dependency group
    uv_extras=["serving"],      # Include the "serving" optional extra
)
```

For example, given this `pyproject.toml`:

```toml
[project]
name = "my-ml-project"
dependencies = ["mlflow", "scikit-learn", "numpy"]

[project.optional-dependencies]
serving = ["flask", "uvicorn"]

[dependency-groups]
ml = ["xgboost", "lightgbm"]
```

The resulting `requirements.txt` will contain all base dependencies plus `xgboost`, `lightgbm`, `flask`, and `uvicorn` with their full transitive graphs, all pinned to exact lockfile versions.

The following table summarizes the three parameters:

| Parameter         | Type            | What it does                                               |
| ----------------- | --------------- | ---------------------------------------------------------- |
| `uv_project_path` | `str` or `Path` | Path to the uv project directory containing `uv.lock`      |
| `uv_groups`       | `list[str]`     | Dependency groups to include (maps to `uv export --group`) |
| `uv_extras`       | `list[str]`     | Optional extras to include (maps to `uv export --extra`)   |

All three parameters are available on both `mlflow.pyfunc.log_model()` and `mlflow.pyfunc.save_model()`.

## How it works

The integration touches two sides of the model lifecycle: logging and restoring.

### Logging models

When `log_model()` or `save_model()` is called, MLflow checks whether a uv project exists in the working directory (or the path specified by `uv_project_path`). Detection requires both `uv.lock` and `pyproject.toml` to be present. If found, MLflow runs:

```
uv export --no-dev --no-hashes --frozen --no-header --no-emit-project --no-annotate
```

This produces a clean, pip-compatible requirements list with exact versions from the lockfile. The `--frozen` flag ensures the lockfile is used as-is without triggering a re-resolve.

If `uv_groups` or `uv_extras` are specified, the corresponding `--group` and `--extra` flags are appended to the command.

### Loading model

When a model with a `uv.lock` artifact is loaded, MLflow uses `uv sync --frozen --no-dev` to recreate the environment from the lockfile. This is faster than pip-based installation because uv resolves and installs dependencies in parallel, and the lockfile eliminates the need for resolution entirely.

If uv is not available on the deployment machine or `uv sync` fails, MLflow raises an error instead of silently falling back to pip. This is intentional: a silent fallback would install unpinned dependencies without hash verification, defeating the security guarantees of the lockfile. To deploy models logged with uv, make sure uv is installed in the target environment.

## Environment variables

Two environment variables provide fine-grained control:

| Variable                | Default | What it does                                                           |
| ----------------------- | ------- | ---------------------------------------------------------------------- |
| `MLFLOW_UV_AUTO_DETECT` | `true`  | Set `false` to disable auto-detection of uv projects                   |
| `MLFLOW_LOG_UV_FILES`   | `true`  | Set `false` to skip saving `uv.lock` and `pyproject.toml` as artifacts |

When `MLFLOW_UV_AUTO_DETECT` is `false`, MLflow skips uv detection entirely and uses standard pip-based inference. An explicit `uv_project_path` parameter overrides this setting.

When `MLFLOW_LOG_UV_FILES` is `false`, MLflow still uses `uv export` to populate `requirements.txt`, but does not copy the lockfile and project files as artifacts. This is useful when lockfile size is a concern and you do not need `uv sync` restoration.

## Getting started

**Prerequisites:** [uv](https://docs.astral.sh/uv/) version 0.5.0 or later.

If you already have a uv-managed project, upgrade to the latest MLflow and log a model.

```bash
pip install 'mlflow>=3.11'
```

Then log a model from within your uv project:

```python
import mlflow

mlflow.pyfunc.log_model(
    name="my-model",
    python_model=my_model,
)
# Check the artifacts: requirements.txt now contains pinned lockfile deps
# uv.lock and pyproject.toml are saved alongside the model
```

For a complete, runnable example demonstrating auto-detection, explicit paths, groups, and extras, see the [example project on GitHub](https://github.com/mlflow/mlflow/tree/master/examples/uv-dependency-management).
